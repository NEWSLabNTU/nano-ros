## ROOT CAUSE FOUND + FIXED (2026-06-15) — `CARGO_PROFILE_RELEASE_LTO=off`

**It is the workflow's `CARGO_PROFILE_RELEASE_LTO: "off"` env, not the CI image.**
`host-integration-tests.yml`'s two fixture-build steps set
`CARGO_PROFILE_RELEASE_LTO=off` (a disk/speed trim, scoped to the steps). That env
**overrides the generated FFI crate's `[profile.release] lto = true`** — and the FFI
crate sets `lto = true` *precisely* so fat LTO DCE-strips its redundant unwinding
`std` (incl. the global `rust_begin_unwind`) that it never calls (it is
`panic="abort"`). With LTO forced off, that `std` panic runtime is retained →
collides with `libnros_cpp.a`'s copy → `multiple definition`. The earlier "not
reproducible locally" was simply because dev/container builds used the FFI crate's
own `lto=true`.

**Reproduced + bisected locally** (build the FFI crate, `nm | grep rust_begin_unwind`):

| `CARGO_PROFILE_RELEASE_LTO` | `rust_begin_unwind` |
|---|---|
| `true` (FFI's own profile, dev default) | 0 (stripped) |
| `thin` | 1 (retained — thin LTO does NOT strip it) |
| `off` (CI) | 1 (retained → the dup) |

**Fix:** drop `CARGO_PROFILE_RELEASE_LTO: "off"` from the **Build workspace
fixtures** step (kept on **Build rust core fixtures** — those are Rust *binaries*,
one `std`, no dup). **Validated:** a clean `workspace-fixtures-build.sh native
{cpp,mixed}` under the exact CI env minus the override (`RUSTFLAGS=-C debuginfo=0`,
`NROS_BUILD_JOBS=2`, `CARGO_BUILD_JOBS=2`) links both `native_entry` binaries.
Confirm on the next CI run, then archive.

Long-term hardening (optional, lto-independent): fold the FFI interface crates into
a single `std`-bearing staticlib (see "Direction" below) so correctness no longer
relies on fat-LTO DCE. The original investigation that led here follows.

The native **C++** and **mixed C/C++/Rust** example-workspace Entry pkgs
(`examples/workspaces/{cpp,mixed}/src/native_entry`) fail to link **on CI**
(`host-integration-tests`, ubuntu-22.04). Surfaced 2026-06-15 while triaging
[issue 0057](archived/0057-host-integration-tests-red-oom-and-skip-gating.md):
after the phase-249 P4a register-stub fix landed (commit `fa2ecb60a`), the three
`workspace_metadata` cpp/mixed `*_is_prebuilt` / `*_starts_prebuilt_runtime` tests
went from "register stub undefined" to a **second, deeper** link failure. Still
live on `main` (the 0057 close validated a CI-equivalent set that did not exercise
these cpp/mixed workspace fixtures).

## Symptom (CI)

```
/usr/bin/ld: nano_ros/packages/core/nros-cpp/libnros_cpp.a(std-46ee58edabb6fcaa.std.…-cgu.0.rcgu.o):
  in function `__rustc::rust_begin_unwind':
…/library/std/src/panicking.rs:622: multiple definition of `__rustc::rust_begin_unwind';
  src/talker_pkg/nano_ros_cpp_ffi_std_msgs/target/release/libnano_ros_cpp_ffi_std_msgs.a(…cgu.0.rcgu.o): first defined here
```

## Root cause

The native C/C++ Entry link line carries **two Rust staticlibs that each bundle
`std`**:

- `libnros_cpp.a` — the C++ umbrella (`crate-type=["staticlib"]`).
- `libnano_ros_cpp_ffi_<pkg>.a` — the per-message-package FFI shim, generated by
  `nros_generate_interfaces()`, also `crate-type=["staticlib"]`.

Each `staticlib` crate statically bundles its own `std`, so both define the
`#[panic_handler]` lang item `rust_begin_unwind`; GNU ld rejects the second as
`multiple definition`. This **violates the documented design invariant**
(`packages/core/nros-cpp/CMakeLists.txt` ~L148): *"the C++ umbrella is the ONLY
Rust staticlib on a C++ link line ⇒ std once ⇒ no `--allow-multiple-definition`."*
Generated message packages add extra Rust staticlibs the invariant didn't account
for.

For **Rust-node** workspaces the per-configure runtime umbrella
(`nros_synth_runtime_umbrella`, `cmake/NanoRosRuntimeCrate.cmake`) bundles
everything into one staticlib (one `std`) — but it does **not** fold in the
generated FFI interface crates, and it is a **no-op for pure-C/C++** workspaces
(no Rust node), so the cpp / mixed Entry never gets the single-`std` treatment.

## Not reproducible locally (the unresolved part)

A clean local `scripts/build/workspace-fixtures-build.sh native {cpp,mixed}`
**links cleanly** — same rustc (`ac68faa20c58…`, 1.96.0, v0 mangling), same GNU ld
2.38. `nm` on the local archives shows the FFI staticlib and `libnros_cpp.a` carry
**no `rust_begin_unwind` at all**, while CI's both emit it (same `std` cgu hash).
Identical toolchain + identical generated sources, materially different staticlibs.

Investigated and **ruled out**:

- **FFI `panic="abort"` profile being dropped.** The FFI crate's `Cargo.toml`
  carries `[profile.release] { opt-level="s", lto=true, panic="abort" }`, and
  `[profile.*]` is ignored for a *workspace-member* crate — but the generated
  template (`cmake/cpp_ffi_Cargo.toml.in`) **already opens with an empty
  `[workspace]` table**, so the crate is its own root and the profile *is* honored
  on both dev and CI. `cargo metadata` confirms a single-member workspace locally.
  So the divergence is **not** profile-honoring. (Forcing the FFI crate to
  `panic="unwind"` errors outright — strategy conflict with its precompiled deps —
  confirming it is built `panic="abort"`.)

**Still open:** why CI's `panic="abort" lto=true` FFI staticlib *retains* a global
`rust_begin_unwind` and every standalone build strips it.

### Repro matrix (2026-06-15) — narrowed to the CI image

Built the generated FFI crate (`cargo build --release`, profile honored) and `nm`'d
the staticlib in three environments — all produce **0** `rust_begin_unwind`:

| env | rustc | ld | `rust_begin_unwind` |
|-----|-------|----|--------|
| dev host (newer kernel) | 1.96.0 `ac68faa20` | GNU ld 2.38 | 0 |
| `ros:humble-ros-base-jammy` (ubuntu 22.04 = CI base OS) | 1.96.0 `ac68faa20` | GNU ld 2.38 | 0 |
| full local cmake/corrosion workspace build (`workspace-fixtures-build.sh native cpp`) | same | same | links clean |

So **toolchain, OS, binutils, the `[profile] panic="abort"+lto`, and the corrosion
cmake path are all RULED OUT** — every reproduction outside CI honors the abort
profile and DCE-strips the unwinding handler. The symbol only appears in the
**`ghcr.io/newslabntu/nano-ros-ci:humble`** CI container, whose build environment
(a cargo config / env var / corrosion or rust setup that drops the FFI crate's
`[profile.release]` → `panic="unwind"`, no LTO) is the remaining variable. That
image is **private** (ghcr `unauthorized` without creds), so it could not be pulled
to reproduce.

**Unblock:** pull `nano-ros-ci:humble` (needs `docker login ghcr.io`) and inside it
`cargo build --release` the FFI crate + `nm | grep rust_begin_unwind`, OR add a
one-shot CI step that `nm`s the FFI staticlib. Confirm it built `panic="unwind"`
there; the fix then is to force the panic strategy / single-`std` explicitly in the
corrosion build command (env-independent), not via the per-crate `[profile]`. No
fix can be validated until this image is reproducible.

## Direction (do NOT use `--allow-multiple-definition`)

The mask is explicitly rejected (the footgun the L148 invariant + phase-249
weak-symbol work removed). The deterministic fix is **one `std` on the link line**:

1. **Preferred** — build the generated per-package FFI interface crates as `rlib`
   and fold them into a single umbrella staticlib (extend
   `nros_synth_runtime_umbrella` to cover pure-C/C++ workspaces and to bundle the
   FFI crates), so only one staticlib (one `std`) reaches the linker.
2. Alternatively, force the FFI staticlibs to not retain `std`'s panic runtime
   deterministically on CI (once the LTO/DCE divergence above is understood).

**First reproduce the CI divergence** (a CI-matching container, or instrument the
CI build to `nm` the FFI staticlib) — the fix cannot be verified from a dev box
that already links clean.

## Scope / impact

- Blocks the `workspace_metadata` cpp/mixed prebuilt-Entry tests on the
  host-integration lane (3 of the 4 real failures remaining after issue 0057's
  cap + triage; pure-C and Rust workspaces link fine). The 4th,
  `qos_overrides_runtime_delivery`, is unrelated (phase-250 churn).
- Pure-C++ standalone examples may share this hazard if they link a generated FFI
  staticlib alongside `libnros_cpp.a` — verify whether they pass on CI to scope
  whether this is workspace-specific or general C++-codegen.
