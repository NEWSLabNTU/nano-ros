## Summary (as filed)

Deterministic on the 2026-07-12 sweep: `realtime_tiers_{c,cpp}_nuttx_e2e`
(ctrl counter 4 vs telem 4 — the 10 ms tier not outrunning the 100 ms tier)
and `realtime_subnode_cpp{,_portable}_e2e` (ctrl=5 telem=5 — per-group
`bind_group_sched` seeding suspect). Both shapes published, at the SAME rate.

## Root cause — no scheduling bug; the fixtures ran museum GENERATED TUs

All four lanes (plus `realtime_tiers_rust_nuttx`) **pass on truly-fresh
fixtures**: wipe the workspace `build-*` dirs → rebuild → 5/5 green,
including 5 consecutive nuttx-rust runs (its one failure was a
first-run-after-build cold-start flake).

The "fresh fixtures" of the sweep were stale in a way NO existing guard saw.
The entry TU is generated at **CMake configure time**
(`nano_ros_entry` → `execute_process(nros codegen entry …)`), and two layers
were blind to the codegen **tool**:

1. `CMAKE_CONFIGURE_DEPENDS` covered the launch XML / `system.toml` /
   `package.xml` (the depfile) but NOT the `nros` binary — so the
   `fd32a0f75` group-split fallback and the phase-281 tier emitters never
   re-ran in an existing build dir. The fix commit itself warned
   "regenerating requires wiping the workspace build dir".
2. `workspace-fixture-signature.sh` (both the build stamp and the
   test-time `require_prebuilt_workspace_binary` staleness gate) hashed
   only workspace-local sources — a fixture generated by a months-old
   emitter verified as "fresh" because its *sources* were current. The
   msg-codegen custom commands already depended on the binary; only the
   configure-time entry codegen and the signature did not.

Symptom mechanics: the pre-fallback generated TU put the group-split node
(and, for the museum nuttx TUs, both nodes) on ONE tier's executor, so both
timers advanced once per that executor's spin period — exactly equal
counters.

## Fix (two guards, one per blind layer)

- `cmake/NanoRosEntry.cmake`: the `nros` binary joins
  `CMAKE_CONFIGURE_DEPENDS` — a CLI rebuild makes the next `cmake --build`
  re-run configure and therefore `nros codegen entry`. The writer already
  skips byte-identical output, so no-op tool changes don't churn recompiles
  (verified: touch → "Re-running CMake…" → TU untouched).
- `scripts/build/workspace-fixture-signature.sh`: signature bumped to
  `nros-workspace-fixture-v2` and now hashes the CLI binary's content —
  shared by the build-side stamp and the test-side gate, so a stale-tool
  fixture now FAILS LOUD ("fixture is stale: … Run `just native
  build-workspace-fixtures`") instead of soft-passing. Negative-tested
  (corrupted stamp → FAIL; restored → PASS).

One-time cost: every pre-v2 workspace-fixture stamp reads stale until its
family is rebuilt once.

## Verified

`realtime_subnode_cpp_e2e`, `realtime_subnode_cpp_portable_e2e`,
`realtime_tiers_c_nuttx_e2e`, `realtime_tiers_cpp_nuttx_e2e`,
`realtime_tiers_rust_nuttx_e2e` — 5/5 green in one run under the v2
signature; nuttx-rust additionally 5× serial green.
