# Phase 68 — Alloc-Free C/C++ Bindings & Executor Simplification

**Goal**: Eliminate generics from `Executor`, remove `Box` wrappers from C/C++ FFI layers,
and make the core C/C++ APIs fully alloc-free. `alloc` is only required when the underlying
Rust API itself needs it (parameter services, action callback trampolines).

**Status**: Not Started

**Priority**: High

**Depends on**: None (self-contained refactor)

## Overview

The `Executor` type currently has three generic parameters: `Executor<S, MAX_CBS, CB_ARENA>`.
`S` is the RMW session type (always inferred, never written by users). `MAX_CBS` and
`CB_ARENA` are const generics controlling the callback arena — the only env-var-driven
constants in the entire codebase that use const generics instead of plain `pub const`.

This design forces the C/C++ FFI layers to heap-allocate the executor via `Box` because
C/C++ cannot express Rust const generics. All 41 other build-time size variables (zpico
slots, xrce buffers, smoltcp limits, params capacity) use the consistent pattern of
`build.rs` → `pub const` → fixed-size arrays. The executor is the sole outlier.

Additionally, `nros-node` already depends on the concrete RMW backends via optional
features (`rmw-zenoh`, `rmw-xrce`, `rmw-cffi`), so the `S` parameter provides no
dependency isolation — it's pure abstraction with no architectural benefit on RTOS
targets that run a single executor with a single backend.

### What changes

1. **Remove all generics from `Executor`** — `S` resolved via feature-gated type alias,
   `MAX_CBS`/`CB_ARENA` resolved via env-var-driven `pub const` (matching all other crates)
2. **Remove `Box` from C/C++ executor handling** — use inline opaque storage like
   publishers, subscriptions, and other entities already do
3. **Gate `alloc` only where the Rust API genuinely needs it** — parameter services
   (`Box<dyn ParamServiceProcessor>`), action callback trampolines, timer closures

### Why now

- C/C++ APIs currently require `alloc` to create an executor — blocking use on no_std
  targets without a global allocator
- Entity types (publisher, subscription, service client, guard condition) were already
  converted to inline opaque storage — the executor is the last remaining `Box` in the
  core path
- Simplifies the public API: `Executor::open(&config)` with no turbofish

### 68.1 — Remove const generics from `Executor`

Replace `Executor<S, const MAX_CBS: usize, const CB_ARENA: usize>` with
`Executor<S>` where `MAX_CBS` and `CB_ARENA` become `pub const` from
`nros-node/src/config.rs` (already generated by `build.rs` from env vars).

This matches the pattern used by all 41 other build-time size variables:
`ZPICO_MAX_PUBLISHERS`, `XRCE_MAX_SUBSCRIBERS`, `NROS_MAX_PARAMETERS`, etc.

**Before:**
```rust
pub struct Executor<S, const MAX_CBS: usize = DEFAULT_MAX_CBS,
                       const CB_ARENA: usize = DEFAULT_ARENA_SIZE> {
    arena: [MaybeUninit<u8>; CB_ARENA],
    entries: [Option<CallbackMeta>; MAX_CBS],
    ...
}
// User writes: Executor::<_, 0, 0>::open(&config)
// User writes: Executor::<_, 8, 8192>::open(&config)
```

**After:**
```rust
pub struct Executor<S> {
    arena: [MaybeUninit<u8>; config::ARENA_SIZE],
    entries: [Option<CallbackMeta>; config::MAX_CBS],
    ...
}
// User writes: Executor::open(&config)
// Per-example config via NROS_EXECUTOR_MAX_CBS=0 NROS_EXECUTOR_ARENA_SIZE=0
```

Examples that used `<_, 0, 0>` set env vars in their `.cargo/config.toml` `[env]` section
or justfile recipe. Each example is workspace-excluded with its own build config.

**Files:**
- `packages/core/nros-node/src/executor/spin.rs` — struct definition, all `impl` blocks
- `packages/core/nros-node/src/executor/action.rs` — action `impl` blocks
- `packages/core/nros-node/src/lib.rs` — re-exports
- `packages/core/nros/src/lib.rs` — re-exports, doc comments

### 68.2 — Remove session type parameter `S` from `Executor` and `Node`

Replace `Executor<S>` with `Executor` using feature-gated type aliases in `nros-node`.
`nros-node` already depends on the concrete backends via optional features.

**New type aliases in `nros-node`:**
```rust
// nros-node/src/session.rs (new)
#[cfg(feature = "rmw-zenoh")]
pub(crate) type ConcreteSession = nros_rmw_zenoh::ZenohSession;
#[cfg(feature = "rmw-xrce")]
pub(crate) type ConcreteSession = nros_rmw_xrce::XrceSession;
#[cfg(feature = "rmw-cffi")]
pub(crate) type ConcreteSession = nros_rmw_cffi::CffiSession;

// Public type aliases for entity handles
pub type RmwPublisher = <ConcreteSession as Session>::PublisherHandle;
pub type RmwSubscriber = <ConcreteSession as Session>::SubscriberHandle;
pub type RmwServiceServer = <ConcreteSession as Session>::ServiceServerHandle;
pub type RmwServiceClient = <ConcreteSession as Session>::ServiceClientHandle;
```

**Executor and Node become non-generic:**
```rust
pub struct Executor {
    session: SessionStore<ConcreteSession>,
    arena: [MaybeUninit<u8>; config::ARENA_SIZE],
    entries: [Option<CallbackMeta>; config::MAX_CBS],
    ...
}

pub struct Node<'a> {
    session: &'a mut ConcreteSession,
    ...
}
```

**Entity types lose handle parameter:**
```rust
// Before: EmbeddedPublisher<M, S::PublisherHandle>
// After:  EmbeddedPublisher<M>
pub struct EmbeddedPublisher<M> {
    handle: RmwPublisher,
    _phantom: PhantomData<M>,
}
```

The `Session` trait remains in `nros-rmw` for backend implementors. Only the user-facing
types in `nros-node` become concrete.

When no RMW feature is active, the executor module is `#[cfg]`-gated out (it already is
for `open()` methods). A `compile_error!` can warn if the user tries to use `Executor`
without an RMW backend.

**Files:**
- `packages/core/nros-node/src/session.rs` — NEW: feature-gated type aliases

- `packages/core/nros-node/src/executor/handles.rs` — remove handle type params from
  `EmbeddedPublisher`, `Subscription`, `EmbeddedServiceServer`, `EmbeddedServiceClient`,

### 68.3 — Remove `Box` from nros-c executor

Replace `_internal: *mut c_void` (Box-allocated `CExecutor`) with inline opaque storage
in `nros_executor_t`, using the same pattern as `nros_support_t`, `nros_publisher_t`, etc.

After 68.1–68.2, `Executor` has a fixed size per build. `build.rs` computes the storage
constant and emits it to both Rust (`config.rs`) and C (`nros_config_generated.h`).

**`nros_executor_t` changes:**
```c
// Before:
typedef struct nros_executor_t {
    ...
    void* _internal;  // Box<CExecutor>
} nros_executor_t;

// After:
typedef struct nros_executor_t {
    ...
    _Alignas(8) uint8_t _opaque[NROS_EXECUTOR_STORAGE_SIZE];
} nros_executor_t;
```

**Rust side:**
- `nros_executor_init()`: `core::ptr::write()` into `_opaque` instead of `Box::new()`
- `nros_executor_fini()`: `core::ptr::drop_in_place()` instead of `Box::from_raw()`
- `get_executor()`: cast `_opaque.as_mut_ptr()` instead of dereferencing `_internal`
- Remove `#[cfg(feature = "alloc")]` gates from executor init/fini/spin

**Size computation in `build.rs`:**
```rust
// Upper bound formula:
// arena + entries + session + metadata + padding
let arena_size = max_handles * message_buffer_size * 2 + 4096;
let entries_size = max_handles * 64;  // sizeof(Option<CallbackMeta>)
let overhead = 1024;  // SessionStore + heapless strings + enums + Arc + alignment
let executor_opaque_u64s = (arena_size + entries_size + overhead + 7) / 8;
```

Compile-time assertion in Rust verifies the computed size is sufficient.

**Files:**
- `packages/core/nros-c/build.rs` — compute and emit `EXECUTOR_OPAQUE_U64S`, generate
  `nros_config_generated.h` with `NROS_EXECUTOR_STORAGE_SIZE`
- `packages/core/nros-c/src/constants.rs` — add `EXECUTOR_OPAQUE_U64S`
- `packages/core/nros-c/src/executor.rs` — replace `Box` with inline opaque storage,
  remove `#[cfg(feature = "alloc")]` from CExecutor/get_executor/init/fini/spin
- `packages/core/nros-c/include/nros/executor.h` — replace `void* _internal` with
  `uint8_t _opaque[NROS_EXECUTOR_STORAGE_SIZE]`, include generated config header

### 68.4 — Remove `Box` from nros-cpp-ffi executor

Replace `Box<CppContext>` with inline opaque storage in the C++ `Executor` class,
matching the pattern already used for publisher, subscription, etc.

**Rust side (`nros-cpp-ffi/src/lib.rs`):**
- `nros_cpp_init()` → write `CppContext` into caller-provided storage via `ptr::write()`
- `nros_cpp_fini()` → `ptr::drop_in_place()` on the storage
- `nros_cpp_spin_once()` → cast storage pointer
- Remove `#[cfg(feature = "alloc")]` gates from init/fini
- Add `CPP_EXECUTOR_OPAQUE_U64S` constant and compile-time assertion

**C++ side (`nros-cpp/include/nros/executor.hpp`):**
```cpp
// Before:
class Executor {
    void* handle_;  // Box<CppContext>
};

// After:
class Executor {
    alignas(8) uint8_t storage_[NROS_CPP_EXECUTOR_STORAGE_SIZE];
    bool initialized_;
};
```

**Files:**
- `packages/core/nros-cpp-ffi/build.rs` — add config generation, compute and emit
  `CPP_EXECUTOR_OPAQUE_U64S`
- `packages/core/nros-cpp-ffi/src/lib.rs` — replace `Box` with inline storage for
  `CppContext`, remove `alloc` gates from init/fini/spin

## Notes

- **Entity handle const generics** (`Subscription<M, RX_BUF>`, `EmbeddedServiceServer<Svc, REQ_BUF, REPLY_BUF>`) are NOT removed in this phase. These are per-entity buffer sizes chosen at the call site, not build-wide configuration. They serve a different purpose from the executor arena.
- **The `Session` trait** in `nros-rmw` is unchanged. Backend implementors still implement `Session`. Only the user-facing types in `nros-node` become concrete.
- **Backward compatibility**: This is a breaking change to the Rust public API. The `nros` crate's re-exports change signature. A migration guide should note: remove turbofish, set env vars for non-default sizes.
- **nros-c service server** (`nros_service_t`) has a `_internal` pointer for the server handle that was not converted in the earlier Box-removal work (only the client was converted). Item 68.5 addresses this.
- **Action `Box` usage**: Action server/client in both nros-c and nros-cpp-ffi use `Box` for callback trampoline state (`ActionServerInternal`, `ActionClientInternal`). These are intentionally kept — the Rust action API itself uses `alloc` for type-erased callback storage, so requiring `alloc` for actions is consistent.
- **Parameter services**: `nros-node` uses `Box<dyn ParamServiceProcessor>` for type erasure in the executor spin loop, and `Box<Response>` for parameter service responses (large heapless arrays that would overflow the stack). This is gated on `param-services` feature and unrelated to FFI.
- **Timer closures**: `TimerCallback = Box<dyn FnMut() + Send>` requires `alloc`. The C API timer uses function pointer + context (no Box needed). The C++ API timer uses the handle ID pattern (already alloc-free).
