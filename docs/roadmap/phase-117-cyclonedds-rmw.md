## Background

### Why Cyclone DDS

`autoware-safety-island` (Arm Limited) uses vendored upstream Cyclone DDS (commit `f7688ce7`, ~0.11.x) on Zephyr Cortex-A/R targets. nano-ros pins to **tag `0.10.5`** instead — matches the installed `ros-humble-cyclonedds` 0.10.5 + `ros-humble-rmw-cyclonedds-cpp` 1.3.4 packages, ensuring wire-compat with stock ROS 2 Humble for E2E tests. App-side wrapper at `actuation_module/include/common/dds/{dds,publisher,subscriber,config}.hpp` uses **15 Cyclone C symbols + 4 types** — minimal embedded surface. Static config via `dds_create_domain_with_rawconfig` (no XML parser).

`dust-dds` (current `rmw-dds` backend) is heavier and Rust-native; mismatches Autoware contributors' C++ workflow. Cyclone is mature, ROS 2 Tier-1 (`rmw_cyclonedds_cpp` upstream), and battle-tested on safety MCUs.

### Why a C++ backend (not Rust)

`packages/core/nros-rmw-cffi/include/nros/rmw_vtable.h` already defines a C ABI for custom RMW backends:

```c
typedef struct nros_rmw_vtable_t {
    nros_rmw_ret_t (*open)(...);
    nros_rmw_ret_t (*create_publisher)(...);
    nros_rmw_ret_t (*publish_raw)(...);
    /* ... ~20 fn pointers ... */
} nros_rmw_vtable_t;

nros_rmw_ret_t nros_rmw_cffi_register(const nros_rmw_vtable_t *vtable);
```

Phase 102 closed this surface. A pure-C++ backend implementing the vtable + calling `nros_rmw_cffi_register` at init satisfies the runtime — no Rust shim, no bindgen, no FFI layering on top of FFI. Mirrors Autoware's existing `common/dds/*.hpp` style verbatim, lowering review cost for downstream contributors.

### Type descriptor problem

Cyclone needs `dds_topic_descriptor_t` per topic type, generated by `idlc` from `.idl`. nano-ros runtime passes only `type_name` + `type_hash` strings via the vtable. Bridge: extend `cargo nano-ros` codegen to emit Cyclone descriptor `.c/.h` alongside existing nros message structs, with self-registering static-init that builds a `type_name → descriptor*` table the backend looks up in `create_publisher` / `create_subscriber`.

### Stock ROS 2 wire compatibility

Cyclone is strongly typed at the `dds_create_topic` boundary: every topic carries a descriptor, and Cyclone matches publishers↔subscribers by `(topic_name, type_name)` pair. To interoperate with stock `rmw_cyclonedds_cpp` we need the **same** topic name and **same** type name on both sides. Stock RMW derives both via fixed conventions:

| Surface | Stock convention |
|---|---|
| Pub/sub topic name  | user `<topic>` → `rt/<topic>` |
| Service Request topic | user `<svc>` → `rq/<svc>Request` |
| Service Reply topic   | user `<svc>` → `rr/<svc>Reply` |
| Message type name   | `.msg` `<pkg>/msg/<Type>` → IDL `<pkg>::msg::dds_::<Type>_` |
| Service Request type | `.srv` → IDL `<pkg>::srv::dds_::<Svc>_Request_` with leading `cdds_request_header_t` field |
| Service Response type | `.srv` → IDL `<pkg>::srv::dds_::<Svc>_Response_` with leading `cdds_request_header_t` field |
| Service request-id  | inline `cdds_request_header_t { writer_guid[16]; sequence_number }` as the first field of the typed payload |
| Service QoS default | `rmw_qos_profile_services_default` = reliable + transient_local + keep_last(10) |

The interim 117.7.B `ServiceEnvelope` approach matches **none** of these conventions for services (it uses a backend-defined wrapper type instead of inlining the request header into the user's typed Response). 117.X.1–117.X.5 close the gap by integrating `rosidl_adapter`-shaped codegen and adopting upstream's header layout.

### Target platforms

| Autoware target | Class | nano-ros board | Status |
|---|---|---|---|
| `fvp_baser_aemv8r_smp` | Zephyr ARMv8-R Cortex-A SMP (FVP sim) | none | NEW |

### `nros-cpp` parameter API gap

`packages/core/nros-c/include/nros/parameter.h` exposes `nros_parameter_server_t` with bool/i64/f64/string declare/get/set. `packages/core/nros-cpp/include/nros/` lacks a wrapper. Autoware uses `declare_parameter<double>("ctrl_period", 0.15)` (variant-based, node-local, no service exposure) — only API gap blocking nros-cpp adoption. Wrap existing C server under `nros::Node::declare_parameter<T>()` / `get_parameter<T>()`.

**No `Cargo.toml`, no `build.rs`.** Path excluded from Cargo workspace. Pure CMake project. Final artifact `libnros_rmw_cyclonedds.a` + installed CMake config consumable via `find_package(NrosRmwCyclonedds REQUIRED)`.


### Consumer integration

`nros-cpp` consumers add a CMake option `NROS_CPP_RMW`:
```cmake
set(NROS_CPP_RMW "cyclonedds" CACHE STRING "RMW backend")
if(NROS_CPP_RMW STREQUAL "cyclonedds")
    find_package(NrosRmwCyclonedds REQUIRED)
    target_link_libraries(${target} PRIVATE NrosRmwCyclonedds::NrosRmwCyclonedds)
    target_compile_definitions(${target} PRIVATE NROS_RMW_CYCLONEDDS=1)
endif()
```

`nros_cpp_init()` (in `nros-cpp/src/init.cpp`) calls `nros_rmw_cyclonedds_register()` under `#ifdef NROS_RMW_CYCLONEDDS`. Single-line guard per backend; same pattern future C++ backends reuse.

**No Cargo feature** for `rmw-cyclonedds` in v1. Rust app consumers out of scope (no requestor). If needed later, add `nros-rmw-cyclonedds-sys` shim crate that declares `extern "C" fn nros_rmw_cyclonedds_register()` + locates pre-built lib via `CYCLONEDDS_RMW_INSTALL_DIR` env.

### Type descriptor codegen extension

Extend `cargo-nano-ros` with `generate-cyclonedds-types` subcommand:

1. For each `.msg` in target `package.xml`, derive `.idl` (existing rmw_zenoh path already generates IDL strings).
2. Invoke `idlc` (Cyclone's IDL compiler, in-tree at `cyclonedds/src/tools/idlc/`) → emit `<type>_cyclone.{c,h}`.
3. Emit static-init self-registration:
   ```c
   __attribute__((constructor))
   static void register_<type>(void) {
       nros_rmw_cyclonedds_register_descriptor(
           "std_msgs/msg/Header", &<type>_desc);
   }
   ```
4. Bundle generated artifacts under `packages/codegen/interfaces/<pkg>/cyclonedds/`.

Backend-side registry: `std::map<std::string_view, const dds_topic_descriptor_t*>` (heap on POSIX, `etl::flat_map` on no_std targets). Lookup in `create_publisher` / `create_subscriber`; `NROS_RMW_RET_TYPE_NOT_REGISTERED` on miss.

### Feature flag wiring

`nros-node/Cargo.toml`:
```toml
[features]
rmw-cyclonedds = ["nros-rmw-cyclonedds"]
```
Mutual exclusion with existing `rmw-{zenoh,xrce,dds,uorb,cffi}` enforced via `compile_error!` in `nros-node/src/lib.rs` (extend existing axis check).

`nros-node::start()` under `#[cfg(feature = "rmw-cyclonedds")]` calls `nros_rmw_cyclonedds_register()` extern fn before any session creation.

### Stock ROS 2 interop (117.X)

The phase-117 goal is **full wire compatibility with stock `rmw_cyclonedds_cpp`** so a nano-ros node can publish to a topic that an `rclcpp` subscriber reads (and the reverse), and so service calls cross the stack in either direction. The current state ships nano-ros↔nano-ros pub/sub + services on Cyclone but does not match the upstream wire conventions on either side. The 117.X sub-phases close that gap.

Each fix is independent enough to land separately; gating sequence is documented under "Sequencing" below.

- [x] **117.X.1 — `rosidl_adapter` codegen integration (shell-out path).** New script `scripts/cyclonedds/msg_to_cyclone_idl.py` drives ROS 2's stock `rosidl_adapter` (`/opt/ros/humble/lib/rosidl_adapter/{msg2idl,srv2idl}.py`) on each `.msg` / `.srv`, then post-processes the produced IDL to inject `module dds_ { … }` around every struct and append `_` to the struct name. Result: `idlc -t -l c <output>.idl` produces `dds_topic_descriptor_t::m_typename = "<pkg>::msg::dds_::<Type>_"` / `"<pkg>::srv::dds_::<Type>_{Request,Response}_"` — bit-exactly what stock `rmw_cyclonedds_cpp` emits. New CMake helper `nros_rmw_cyclonedds_generate_from_msg(<output_var> PKG_NAME <name> PKG_DIR <pkg> INTERFACES <list> [OUTPUT_DIR <dir>])` chains the two stages: drives `msg_to_cyclone_idl.py` per file, hands each generated `.idl` to `nros_rmw_cyclonedds_idlc_compile()` with the matching `TYPE_NAME` (`.msg`) or `TYPE_NAMES` (`.srv` → both Request + Response). Helper installs the converter to `share/nros-rmw-cyclonedds/` so `find_package(NrosRmwCyclonedds)` consumers find it via the config's `find_program` walk. `from_msg_smoke` CTest exercises the full path on a synthetic `std_msgs/msg/String` and asserts the registry hands back a descriptor with `m_typename = "std_msgs::msg::dds_::String_"`. Soft-degrades when `rosidl_adapter` isn't installed (skips the test, rest of the harness still passes). The Rust-port alternative (no `/opt/ros/humble` dependency) is tracked separately and not gating Phase 117.
  - **Files:** `scripts/cyclonedds/msg_to_cyclone_idl.py`, `packages/dds/nros-rmw-cyclonedds/cmake/NrosRmwCycloneddsTypeSupport.cmake` (`find_program` for the converter + `nros_rmw_cyclonedds_generate_from_msg` function), `packages/dds/nros-rmw-cyclonedds/CMakeLists.txt` (install the converter), `packages/dds/nros-rmw-cyclonedds/tests/{from_msg_smoke.cpp,types/std_msgs/{package.xml,msg/String.msg}}`, `tests/CMakeLists.txt`.

- [x] **117.X.2 — Topic-name prefix conventions.** Backend `publisher_create` / `subscriber_create` prepend `rt/`; `service_server_create` / `service_client_create` produce `rq/<svc>Request` and `rr/<svc>Reply`. Implemented via header-only `src/topic_prefix.hpp` (`apply()`, `already_prefixed()`, `skip_via_env()`). Idempotent — names already starting with `r[trqr]/` pass through unchanged. Opt-out via `NROS_RMW_CYCLONEDDS_SKIP_PREFIX=1` env so raw-DDS deployments (Autoware safety-island, etc.) keep their own naming. New `topic_prefix_test` CTest covers all four cases (apply / idempotent / env-skip / overflow).
  - **Files:** `src/topic_prefix.hpp`, `src/publisher.cpp` (rt/ prefix), `src/subscriber.cpp` (rt/ prefix), `src/service.cpp` (`service_topic_name()` helper produces `rq/<svc>Request` / `rr/<svc>Reply`), `tests/{topic_prefix_test.cpp,CMakeLists.txt}`. Existing service_smoke / roundtrip / concurrent tests adjusted to use plain service names (not pre-prefixed) so the new prefix logic is exercised end-to-end.

- [ ] **117.X.3 — Replace `ServiceEnvelope` with `cdds_request_header_t`.** Drop `src/types/service_envelope.idl`. Codegen (117.X.1) emits `_Request_` / `_Response_` IDLs with `cdds_request_header_t { writer_guid[16]; sequence_number }` as the **leading field**. Backend's data-path reads `(seq_out)` from that header on take, re-attaches the matching `(writer_guid, seq)` on `send_reply` from the slot table; client filters replies on the same pair. Concurrent-call correlation logic from 117.7.B is preserved verbatim, only the wire location of the header changes.

## Notes / Risks

### R1 — Cyclone-on-Zephyr build
Cyclone CMake assumes POSIX threads/sockets. Zephyr `CONFIG_POSIX_API=y` covers most surfaces; edge cases possible (e.g. `pthread_attr_setstack`). **Mitigation:** Autoware tree already runs Cyclone on the same Zephyr boards — copy their build flags verbatim from `actuation_module/CMakeLists.txt`.

### R2 — Multicast SPDP on Zephyr ethernet
Discovery needs IGMP. Zephyr eth stack supports it but board-specific. **Mitigation:** Cyclone supports unicast-only via `<Discovery><Peers>` (Autoware uses `cfg_peer` static peer list — same path).

### R3 — Type descriptor codegen pipeline
`idlc` is C; needs to compile + invoke from build pipeline. **Mitigation:** in-tree at `cyclonedds/src/tools/idlc/`; build alongside Cyclone in 117.1. Validate generated `.c` compiles in `no_std` consumer crates (no libc deps).

### R4 — R52 has no MMU
Cyclone uses dynamic alloc for entities, internal queues. Sized heap config + monitor for OOM in long-running E2E. **Mitigation:** Autoware tree already runs on R52 — heap budget known. Document required `CONFIG_HEAP_MEM_POOL_SIZE` floor in board crate.

### R5 — ARMv8-R SMP and `Send`/`Sync`
Cyclone spawns its own RX threads; `nros-rmw` shim must be `Send + Sync` across SMP cores. Already a constraint for zenoh backend — reuse `Mutex<>` patterns. Listener trampoline writes to `Activator` ready-set under atomic.

### R6 — Stock ROS 2 wire compat (full path)
Cyclone uses raw DDS topic names — different from `rmw_zenoh`'s `<domain>/<topic>/<type>/TypeHashNotSupported` encoding. **Cyclone backend targets full wire-compat with stock `rmw_cyclonedds_cpp`** — pub/sub, services both. Achieving it requires four upstream-shape conventions:
- topic-name prefixes (`rt/` / `rq/` / `rr/`) — 117.X.2
- type-name mangling (`<pkg>::msg::dds_::<Type>_`) — 117.X.1 / 117.X.4
- in-payload `cdds_request_header_t` for services — 117.X.3
- `rmw_qos_profile_services_default` (reliable + transient_local + keep_last(10)) — 117.X.5

The interim `ServiceEnvelope` approach from 117.7.B is **not** wire-compatible; 117.X.3 replaces it. Validation gates the entire chain on 117.12 E2E. No cross-RMW bridging in scope (see Phase 104).

### R8 — `rosidl_adapter` integration choice
117.X.1 has two viable implementation paths (shell-out to ROS 2's Python rosidl_adapter, vs in-tree Rust mapping). Shell-out is faster to land but adds a host ROS 2 install requirement at codegen time; in-tree is more portable but ~3× the implementation effort. **Mitigation:** start with shell-out; gate the Rust port on a separate phase if the install dependency becomes a problem.

### R9 — XTypes type-info segfault
Cyclone 0.10.5 idlc emits truncated output when type-info section is requested (the issue that forced `-t` in 117.2). Blocks 117.X.6. **Mitigation:** keep 117.X.6 deferred; revisit when bumping the Cyclone pin past 0.10.5 or carrying a patch.

### R7 — Phase 108 events
Cyclone supports liveliness / deadline / message-lost events natively (`dds_set_listener` + status callbacks). v1 leaves `register_*_event` as NULL (`UNSUPPORTED`) per Phase 108 backend-wiring sub-phase plan. Wiring in follow-up phase reusing existing Cyclone listener infrastructure.
