nano-ros · rfc-0015 · rt orchestration

Model 1
one execution model
every RTOS, every language.

A ROS 2 graph — nodes, timers, subscriptions, services — lowered onto an RTOS the same way whether written in Rust, C, or C++. The kernel's fixed-priority scheduler does the real-time work.

the whole model

Three rules. That's it.

01 · tier → task
One RTOS task per priority tier
Not per node, not per callback — so a tier costs exactly one OS priority slot.
02 · task → executor
Each task owns one Executor
All executors run over one shared RMW session per binary — the "Borrowed" store. No N× RAM.
03 · gating
active_groups filter
Every task runs the same setup; the filter admits only its tier's callback groups.

vocabulary — read this first

A task is a thread, not a callback

Four terms from three worlds. The confusing one: task means OS thread, never a unit of work. One task runs many callbacks.

why not 1 thread per callback
That would burn one OS priority slot per callback — Cortex-M0+ has ~4 total. Model 1 bundles callbacks by priority need into one thread per tier; the thread cooperatively runs all its callbacks. Kernel preempts between tiers, executor schedules within.
step 1 — author

A node labels its callback group

Same shape in every language: create a node, tag it with a group name, wire timers/pubs. The group name is the join key to the deployment config.

ctrl_pkg/src/lib.rsrust
fn register(ctx: &mut NodeContext) -> NodeResult<()> {
    let mut node = ctx.create_node(NodeOptions::new("control_node"))?;
    node.callback_group("ctrl")?;               // ← group label
    let pub_ctrl = node.create_publisher_for_topic::<Int32>("/ctrl")?;
    let _t = node.create_timer_for_callback_name(
                 "on_ctrl", TimerDuration::from_millis(10))?;
    node.callback_for_name("on_ctrl").publishes_entity(&pub_ctrl)?;
    Ok(())
}

void Ctrl::on_tick() {

note
The node says nothing about priorities or threads. It only names a group. Where that group runs is a deployment decision — next slide.
step 2 — deploy config

system.toml maps group → tier

One manifest binds each node's group to a named tier, and defines each tier's priority + spin period. This is the deployment SSoT (RFC-0047).

demo_bringup/system.tomlverbatim · ws-realtime-rust
[[component]]
pkg = "ctrl_pkg"
class = "ctrl_pkg::Control"
name = "control_node"
group_tiers = { ctrl = "high" }          # group → tier binding

[[component]]
pkg = "telem_pkg"
name = "telem_node"
group_tiers = { telem = "low" }

[tiers.high]
spin_period_us = 1000
[tiers.high.posix]  priority = 80
[tiers.high.zephyr] priority = 5       # lower = higher on Zephyr
[tiers.high.nuttx]  priority = 110

[tiers.low]
spin_period_us = 10000
[tiers.low.posix]   priority = 10
key idea
Tiers are declared per RTOS in sub-tables. Same nodes, same groups — retarget by picking which sub-table the codegen reads. (See the deploy slide.)
step 3 — entry

The binary entry is one line

Codegen reads the manifest, resolves the tier table, and emits run_tiers. The author writes only the launch reference.

native_entry/src/main.rsrust
nros::main!(launch = "demo_bringup");
native_entry/src/main.cppc++
#include <nros/main.hpp>

NROS_MAIN(::nros::board::NativeBoard,
          "demo_bringup:system.launch.xml");
what it expands to
Both macros resolve the 2-tier table and emit the multi-tier run_tiers entry — one priority task per tier — instead of the single-tier run. All-default-tier collapses back to one task = the shipped simple case.
under the macro

Each tier is a TierSpec + a spin loop

nros-platform · board/tier.rsreal struct
pub struct TierSpec<'a> {
    pub name: &'a str,        // [tiers.<name>]
    pub groups: &'a [&'a str], // active_groups; [] = wildcard
    pub priority: i64,          // raw per-RTOS spawn value
    pub stack_bytes: usize,
    pub spin_period_us: u64,
}
emitted per-tier task bodyshape
run_tiers(TIERS, setup) {
  let shared = open_shared_session();   // once
  for tier in TIERS {
    spawn(tier.priority, tier.stack, || {
      let mut ex = Executor::open_with_session(shared);
      ex.set_active_groups(tier.groups);  // gate
      setup(&mut ex);                     // only its groups take
      loop { ex.spin_once(N); platform_yield(); }
    });
  }
}
inside spin_once
Single thread, non-preemptive, run-to-completion: pump transport once → set one readiness bit per callback with data → dispatch in registration order. One bitmap, no queue. Scheduling policy lives in user space.

one bringup, three deploys
The same nodes + system.toml ship to native, Zephyr, or NuttX by adding a [tiers.*.<rtos>] sub-table and switching the board — no node code changes.
§ how it hits deadlines

Cooperative within · preemptive across

Within a tier: run-to-completion, no mid-callback preemption (no per-callback stacks in no_std). Across tiers: the RTOS preempts a lower tier the instant a higher tier wakes. Real preemption from the kernel, not the executor.

the tradeoff
A 50 ms callback will block a 1 ms-deadline callback if they share a tier. Hard real-time ⇒ put them on different tiers and let the kernel preempt. Cost: a few OS priority slots — never one per callback.

cheap

real-time

Author names a group → system.toml maps it to a tier → codegen emits one RTOS task per tier over one shared session, gated by active_groups. Proven native (C/C++/Rust, ctrl≈5.5× telem); boots on QEMU mps2-an385.
