# ws-bridge-xrce-rust — cross-RMW bridge showcase, XRCE variant (phase-267)

A declarative cross-RMW gateway: a `[[bridge]]` in `system.toml` forwards
`/chatter` from the **zenoh** session to an **XRCE** session (a Micro-XRCE-DDS
Agent) in one process; the agent republishes onto DDS, so a stock ros2/DDS peer
(`ros2 topic echo /chatter`) sees a nano-ros publisher that only speaks zenoh.
**No build.rs, no user bridge code** — plain `nros::main!` + config. The
cyclonedds sibling is [`ws-bridge-rust`](../ws-bridge-rust); xrce differs in that
it is **agent-based** (the egress carries a locator, not a DDS domain) and uses
**lazy type registration** (no descriptor staging / field schema).

## How it works (the clean flow)

1. The node pkg declares what it publishes (`src/talker_pkg/Cargo.toml`):
   ```toml
   [[package.metadata.nros.node.publishes]]
   topic = "/chatter"
   type  = "std_msgs/msg/Int32"
   ```
2. The system declares the bridge, by topic NAME only (`src/demo_bringup/system.toml`):
   ```toml
   [[domain]]
   name = "zen"; rmw = "zenoh"; id = 0
   [[domain]]
   name = "agent"; rmw = "xrce"; id = 0; locator = "udp/127.0.0.1:8888"  # the Agent
   [[bridge]]
   name = "gw"; from = "zenoh:zen"; to = "xrce:agent"
   ```
3. `nros sync` resolves each topic name → its ROS type and writes
   `src/demo_bringup/nros-bridge.toml` (the xrce egress `[[node]]` carries the
   agent `locator`).
4. The Entry is plain `nros::main!(launch = "demo_bringup")`: the macro
   `include_str!`s the generated config and runs `nros_bridge::run_from_config_str`
   — `open_multi` over the two sessions, a `PubSubBridge` per `[[bridge]]`, pump.

## Build & run

```sh
nros sync
cargo build -p native_entry                 # links zenoh + xrce backends
MicroXRCEAgent udp4 -p 8888 &               # the Micro-XRCE-DDS Agent
./target/release/native_entry               # the bridge (dials the agent)
# override endpoints at runtime (phase-267 #113):
#   NROS_BRIDGE_S0_LOCATOR=tcp/127.0.0.1:7447  (zenoh router)
#   NROS_BRIDGE_S1_LOCATOR=udp/HOST:PORT       (agent addr)
#   NROS_BRIDGE_S1_DOMAIN=N                     (the DDS domain the agent joins)
```

  Micro-XRCE-DDS Agent → DDS → a stock ros2 subscriber receives `std_msgs/Int32`
  (8/8). The egress dials the agent at the `[[domain]] agent` locator; the agent's
  DDS participant joins the XRCE-client-requested domain (override with
  `NROS_BRIDGE_S1_DOMAIN`).
- **Lazy registration:** unlike cyclonedds, xrce needs no descriptor staging /
  field schema / typed `register::<M>()` — the macro emits only the backend
  `register()` calls.
