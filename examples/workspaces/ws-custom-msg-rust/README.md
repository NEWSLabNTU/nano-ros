# ws-custom-msg-rust — in-workspace custom message (phase-263 B6)

A minimal product-shaped nano-ros workspace that defines its **own** ROS 2
interface package in-tree and uses it through the ordinary typed pub/sub path.

## Layout

| Path | Role |
| --- | --- |
| `src/custom_msgs/` | interface package — `package.xml` + `msg/Reading.msg` (no Cargo.toml) |
| `src/reading_talker_pkg/` | publishes `custom_msgs/Reading` on `/reading` @1 Hz |
| `src/reading_listener_pkg/` | subscribes `/reading`, echoes the decoded `sequence` on `/reading_seq` |

`Reading.msg`:

```
float64 temperature
float64 humidity
int32 sequence
```

## How the custom message is wired

`custom_msgs` is a **verbatim ROS 2 msg package** (it builds unchanged under
`colcon`). `nros ws sync` walks its `package.xml`, runs the nano-ros codegen
pipeline, and emits a `generated/custom_msgs` Rust crate. The Node pkgs depend on
`custom_msgs = "*"`, resolved to that generated crate via the auto-managed
`[patch.crates-io]` block in `.cargo/config.toml`. The generated `Reading`
implements `RosMessage`, so it flows through `create_publisher_for_topic::<Reading>`
/ `ctx.message::<Reading>()` exactly like a stock `std_msgs` type — the only
difference is the schema is yours.

## Build & run

```sh
nros ws sync                       # codegen generated/custom_msgs (gitignored)
cargo build -p native_entry        # links both Node pkgs + the generated crate
nros plan demo_bringup
```

## Notes

- `generated/` and `Cargo.lock` are gitignored — regenerated by `nros ws sync`.
- Same-process delivery does not occur (issue 0096); a cross-process subscriber
  on `/reading` / `/reading_seq` observes the round-trip (the Track-D assertion).
